# Architecture & Code Quality Analysis

Critical analysis of the codebase's architectural structure, adherence to SOLID/DRY
principles, and file layout. Based on a full read of ~54,800 lines across 137 GDScript files.

---

## 1. Structural Overview

```
scenes/
  world/          — city generation, managers (traffic, police, pedestrian)
    generator/    — 17 chunk builders (4,366 lines)
  vehicles/       — car/boat/helicopter meshes, controllers, AI (12,152 lines)
  player/         — player node, 7 states, camera, weapon, model
  pedestrians/    — NPC model, ragdoll, 3 states
  police/         — officer model, encounter logic
  missions/       — mission marker, marker manager
  ui/hud/         — HUD, minimap, menus
src/
  autoloads/      — 8 singletons (EventBus, GameManager, InputManager, etc.)
  state_machine/  — generic FSM (State + StateMachine, ~72 lines total)
  *.gd            — utility classes (road_grid, city_boundary, biome_map, etc.)
tests/            — 137 GUT test files (one per scene/script under test)
addons/           — GEVP vehicle physics plugin, GUT test framework
```

**Positive**: The directory hierarchy maps closely to game subsystems. The `src/` vs
`scenes/` split separates engine-agnostic logic from scene-coupled code.

---

## 2. SOLID Principle Audit

### Single Responsibility Principle (SRP)

**Violations — grade: C+**

| File | Lines | Too Many Responsibilities |
|------|-------|--------------------------|
| `scenes/world/city.gd` | 1,145 | Chunk loading/unloading, material palette init, all 17 builder instantiation, mesh merging, grid/boundary/biome caching, edge-mismatch repair, fall-through-world detection, disk flush scheduling |
| `scenes/vehicles/car_body_builder.gd` | 1,126 | Body geometry for 5 car variants, window/door/interior meshes, glass shader setup, collision shape creation, door pivot animation data |
| `scenes/vehicles/police_ai_controller.gd` | 840 | Pursuit pathfinding, blockade formation, aggression scaling by wanted level, siren/light bar activation, despawn fade animation |
| `scenes/vehicles/npc_vehicle_controller.gd` | 731 | Road-following AI, intersection decision, collision avoidance, idle behaviour, pedestrian yielding |
| `scenes/world/traffic_manager.gd` | 733 | NPC spawning, despawning, body-variant selection, mesh caching, adaptive spawn-height detection, launch-velocity culling |
| `src/autoloads/mission_manager.gd` | 374 | Mission generation, acceptance, progress tracking, timer management, marker lifecycle, reward distribution |

`city.gd` is the clearest SRP violation: it acts as a world director, asset factory,
streaming loader, and repair tool simultaneously. A World scene with separate
`ChunkStreamer`, `ChunkFactory`, and `WorldRepairer` nodes would distribute these
concerns.

`car_body_builder.gd` mixes data (variant profiles) with mesh construction and
shader configuration. Variant data belongs in a resource file; the builder should
only consume data and emit geometry.

**Well-scoped files (SRP observed)**:
- `src/state_machine/state.gd` (26 lines) — pure interface
- `scenes/vehicles/vehicle_controller.gd` (32 lines) — maps inputs to vehicle properties
- `src/autoloads/day_night_manager.gd` (46 lines) — owns only the time cycle
- `src/autoloads/input_manager.gd` (56 lines) — only context + mouse capture

### Open/Closed Principle (OCP)

**Partial adherence — grade: B−**

The chunk builder system is open for extension: adding a new biome requires a new
`chunk_builder_X.gd` file and one line in `city.gd`. No existing builder changes.
Good.

However, `car_body_builder.gd` contains hard-coded variant dictionaries inside the
script. Adding a new car shape requires editing the builder itself rather than adding
a new `Resource` subclass. A `CarVariantResource` (`@export` fields for profile
points, colours, dimensions) would let artists add variants without touching code.

Mission types (`delivery`, `taxi`, `theft`) are generated by a large `match` block
inside `MissionManager`. Adding a new type requires editing `MissionManager._generate_mission()`.
A `MissionTemplate` resource or strategy object would keep `MissionManager` closed to
modification.

### Liskov Substitution Principle (LSP)

**Observed — grade: A−**

The generic `State` base class (`enter`, `exit`, `update`, `physics_update`,
`handle_input`) is correctly subclassed by all player, pedestrian, and NPC states.
No state violates the contract of its supertype.

`VehicleController`, `BoatController`, and `HelicopterController` share an informal
duck-typed interface (`active`, `physics_update(delta, vehicle)`). Substituting one
for another works without surprises. Formalising this via a common base class or
typed interface would make it explicit.

### Interface Segregation Principle (ISP)

**Partial — grade: B**

GDScript has no formal interfaces, but the project uses duck-typing and narrow
method contracts (e.g., the controller `active`/`physics_update` protocol). The
pattern is close to ISP in spirit.

The problem is `EventBus`: 54 signals in a single autoload means every subscriber
implicitly depends on the entire bus. A subscriber listening to `player_health_changed`
is technically coupled to a node that also knows about `time_of_day_changed`,
`weapon_unlocked`, `mission_failed`, and 50 others. Splitting EventBus into
domain-scoped buses (`PlayerEvents`, `VehicleEvents`, `WorldEvents`) would enforce
narrower dependencies.

### Dependency Inversion Principle (DIP)

**Partially violated — grade: C+**

Several concrete manager classes instantiate their own copies of low-level utilities:

```gdscript
# Duplicated in 7 different files:
var _grid = preload("res://src/road_grid.gd").new()
var _boundary = preload("res://src/city_boundary.gd").new()
```

`TrafficManager`, `PoliceManager`, `PedestrianManager`, `NpcVehicleController`,
`PoliceAiController`, and `MinimapHud` all create private grid/boundary instances.
They should depend on an abstraction provided by injection or a shared singleton,
not on concrete implementations they construct themselves.

## 3. DRY Principle Audit

### Violations

**Grid + boundary instantiation** — 7 files duplicate the same 2–3 lines of
`preload().new()`. A `WorldServices` autoload exposing `grid` and `boundary` as
read-only properties would collapse this to a single source.

**Spawner pattern** — `TrafficManager` and `PedestrianManager` both implement
nearly identical spawn-loop skeletons:

```gdscript
const SPAWN_RADIUS      # same concept, different value
const DESPAWN_RADIUS
const SPAWN_INTERVAL
func _try_spawn() -> void  # same structure: random position, distance checks, height raycast
func _despawn_far() -> void
```

A `SpawnerBase` mixin (or composition object) would unify the retry logic, distance
checks, and despawn sweep, leaving only the payload (what to spawn) as a virtual method.

**Biome-map fetch** — `TrafficManager`, `PoliceManager`, and `PedestrianManager` each
contain an identical 3-line snippet to lazy-fetch the biome map from city metadata.
This belongs in `WorldServices` or the `city.gd` API.

**Fixture creation** — `vehicle_lights.gd` calls `_create_fixture()` with nearly
identical arguments three times (headlights, taillights, reverse lights). The Z-offset
correction differs but the call site is repeated for left and right sides in every group.
The array structure is already there; the offset logic could be a constant table.

**NPC controller overlap** — `npc_vehicle_controller.gd` (731 lines) and
`police_ai_controller.gd` (840 lines) share large sections of road-following logic
(next-waypoint selection, speed-matching by road type, intersection handling). The
police controller is essentially a subtype of the NPC controller with pursuit layered
on top. Inheriting from a common `RoadFollowerBase` would halve the duplicated code.

## 4. Architectural Patterns Assessment

### EventBus (centralized signal hub)

**Strength**: Completely decouples producers from consumers. The HUD never holds a
reference to `WantedLevelManager`; `MissionManager` never references the HUD.
Adding a new listener costs zero changes to the emitter.

**Weakness**: 54 signals in one object creates a monolithic coupling surface. Any
file that connects to one signal implicitly knows the entire bus exists. Signal names
are strings-as-protocol — a typo silently fails at runtime. Splitting by domain
(see ISP above) and adding typed signal wrappers would make misuse compile-visible.

### State Machine

**Strength**: The generic FSM (`state.gd` + `state_machine.gd`, 72 lines combined)
is the project's cleanest abstraction. It is reused identically across player (7),
pedestrian (3), and NPC AI states with zero duplication. Transitions are data-driven
via string names. Lifecycle hooks (`enter`, `exit`, `physics_update`, `handle_input`)
cover all Godot processing moments.

**Weakness**: State nodes receive their context through `get_parent()` calls and
`owner` references rather than dependency injection. A state needing the vehicle
reference calls `get_parent().get_parent()` chains. This makes states fragile to
scene-tree rearrangement and harder to unit-test in isolation.

### Procedural Mesh Generation

**Strength**: Building geometry in code (via `SurfaceTool`) instead of importing
assets gives the infinite world its variety and keeps the repo small. The `_add_quad`
/ `_add_box` helpers are consistently used across all builders.

**Weakness**: The 17 chunk builders have inconsistent internal structure. Some use
class-level constants, some use local `var` declarations; some expose an `init()`
pattern, others don't. A shared `ChunkBuilderBase` class with a typed `build(chunk,
origin_x, origin_z)` signature and common helper methods would enforce consistency
and enable polymorphic dispatch in `city.gd` (replacing 17 named fields with an
`Array[ChunkBuilderBase]`).

### Manager Singletons

Eight autoloaded singletons is on the high side. The split is mostly justified —
each manager owns genuinely global, persistent state — but `GameManager` doubles
as a data store AND a progression system (weapon unlock logic, save/load). Separating
`SaveSystem` and `ProgressionSystem` from `GameManager` would give each a single
job.

`MissionManager` at 374 lines is oversized for an autoload. Mission *generation*,
mission *execution state*, and mission *reward distribution* are three separable
concerns; the current design compresses all three into one class.

### InputManager context switching

**Strength**: Clean and minimal. A single enum guards both mouse capture and
input routing. All input-sensitive systems check `InputManager.current_context`
rather than coupling to each other.

**Weakness**: The VEHICLE context is shared by cars, boats, and helicopters, which
have meaningfully different control schemes. A `VEHICLE_LAND`, `VEHICLE_WATER`,
`VEHICLE_AIR` split would let per-vehicle UI overlays and input hints display
correctly without the driving state needing to detect vehicle type itself.

## 5. File Layout Assessment

### What works well

- `src/state_machine/` isolation — the FSM is engine-agnostic and could be ported
- `tests/` mirrors the source tree one-for-one (each feature file has a test file)
- Autoloads in `src/autoloads/` rather than scattered across `scenes/`
- Chunk builders in a dedicated `generator/` subdirectory

### What needs attention

**`scenes/vehicles/` has 40+ files with no subdirectory grouping.** Suggested split:

```
scenes/vehicles/
  car/          — car_body_builder, car_body_init, car variants
  boat/         — boat_controller, boat_wake
  helicopter/   — helicopter, helicopter_controller, helicopter_body_builder
  common/       — vehicle_controller, vehicle_health, vehicle_lights,
                  vehicle_water_detector, engine_audio, tire_screech_audio
  npc/          — npc_vehicle_controller, police_ai_controller,
                  police_siren, police_light_bar
```

**`scenes/world/` mixes world-management scripts with generator infrastructure.**
`city.gd`, `traffic_manager.gd`, `police_manager.gd`, etc. live alongside
`event_bus.gd` (which is logically an autoload candidate, not a scene script).
Moving generators fully under `generator/` and managers under `world/managers/`
would tighten the layout.

**`src/` utility scripts are flat.** `road_grid.gd`, `city_boundary.gd`,
`biome_map.gd`, `tile_cache.gd`, `tile_resolver.gd`, `river_map.gd` and 8 other
utility classes sit in one directory. A `src/world/` subdirectory for these would
signal their domain.

## 6. Test Architecture Assessment

**Coverage**: 137 test files against ~137 non-test GDScript files is an excellent
1:1 ratio. The GUT framework is used consistently and the `extends GutTest` convention
is enforced by the pre-commit hook.

**Pattern consistency**: Most tests follow a clear `before_each` → instantiate →
`after_each` → free pattern. Source-inspection tests (`script.source_code.contains(...)`)
are used sparingly, only where behaviour cannot be exercised headlessly. Good discipline.

**Gaps observed**:
- Integration tests are absent. Each system is tested in isolation but the full
  crime→heat→wanted→police-spawn pipeline has no end-to-end test.
- Several test files consistently fail to load (`test_vehicle_lights.gd` parse error,
  `test_traffic_manager.gd`, etc.) — pre-existing issues that erode confidence in CI.
- `test_chunk_builder_buildings.gd` tests geometry structure but not biome-dependent
  branching paths.

## 7. Summary Scorecard

| Concern | Grade | Notes |
|---------|-------|-------|
| Single Responsibility | C+ | `city.gd`, `car_body_builder.gd`, AI controllers are over-stuffed |
| Open/Closed | B− | Chunk builders good; variants and mission types hard-coded |
| Liskov Substitution | A− | State machine and controller protocols honoured |
| Interface Segregation | B | Duck-typing is fine; EventBus too wide |
| Dependency Inversion | C+ | Grid/boundary instantiated everywhere instead of injected |
| DRY | C+ | Spawner pattern, grid init, AI road-following duplicated |
| File Layout | B | `scenes/vehicles/` needs subdirectories; rest is clear |
| Test Coverage | B+ | 1:1 ratio, good isolation; some parse errors, no integration tests |
| Overall Architecture | B− | Solid foundations (FSM, EventBus, chunk system); God objects and DRY violations are the main debt |

---

## 8. Prioritised Remediation

1. **[High]** Break `city.gd` into `ChunkStreamer` + `ChunkFactory` + `WorldServices`
   autoload (exposes shared `grid`, `boundary`, `biome_map`). Eliminates 7-file grid
   duplication immediately.

2. **[High]** Extract `SpawnerBase` for traffic and pedestrian managers. Cuts ~150
   lines of duplicated spawn-loop logic.

3. **[Medium]** Introduce `ChunkBuilderBase` with a typed interface. Replace 17
   named fields in `city.gd` with a polymorphic array.

4. **[Medium]** Split `car_body_builder.gd`: move variant profile data to
   `CarVariantResource` files; reduce builder to pure geometry from data.

5. **[Medium]** Split `MissionManager` into `MissionGenerator` + `MissionRunner`.

6. **[Low]** Split `EventBus` into 3–4 domain-scoped buses to narrow coupling.

7. **[Low]** Subdivide `scenes/vehicles/` into `car/`, `helicopter/`, `boat/`,
   `common/`, `npc/` subdirectories.

8. **[Low]** Fix pre-existing test parse errors so the full suite is green.
